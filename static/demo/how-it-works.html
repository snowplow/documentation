01The pipeline

Six stages process every event in real time. Events are kept at atomic grain end-to-end — no sampling, no pre-aggregation.

1. Track

Web, mobile, server, or webhook SDKs emit events to your Collector.

Pipeline in your cloud account · AWS · GCP · Azure

2. Collect

The Collector receives raw payloads over HTTP and persists them.

→ Failed events stream

3. Validate

Every event is checked against its JSON schema in Iglu.

RT Good stream

4. Enrich

Attach geo, user-agent, campaign, currency, and custom API or SQL entities.

5. Load

Stream to warehouse, lake, or forward to downstream tools.

Schema Registry (Iglu) — validation rules live here; Validate fetches per-event.

6. Model

dbt packages turn atomic events into analytics-ready tables.

Latency target: enriched in <1s under normal load.

02Anatomy of a Snowplow event

An event is a JSON object that describes itself. It always carries the schema that defines its shape, so consumers — and your warehouse — always know what they're looking at, even after the definition evolves.

iglu schemav1-0-0
"$schema": "...self-desc/schema/jsonschema/1-0-0",
"description": "A user added a product to their cart",
"self": {
  "vendor": "com.acme",
  "name": "add_to_cart",
  "format": "jsonschema",
  "version": "1-0-0"
},
"type": "object",
"properties": {
  "product_id": { "type": "string" },
  "price":      { "type": "number", "minimum": 0 },
  "currency":   { "type": "string", "enum": ["USD","EUR","GBP"] }
},
"required": ["product_id", "price"]
self-describing eventconforms to schema
{
  "schema": "iglu:com.acme/add_to_cart/jsonschema/1-0-0",
  "data": {
    "product_id": "SKU-9123",
    "price":      49.95,
    "currency":   "USD"
  },
  "contexts": [
    { "schema": "iglu:.../user/1-0-0",
      "data": { "id": "u_42", "tier": "gold" } },
    { "schema": "iglu:.../web_page/1-0-0",
      "data": { "id": "pv_77ab" } }
  ]
}

The contexts array carries entities — reusable building blocks (user, page, product, A/B test, …) that attach to any event. Same entity definition, used everywhere.

03Where events come from

Snowplow provides open-source tracker SDKs for web, mobile, and server-side platforms. All trackers use the same Snowplow tracker protocol, so the Collector accepts events from any of them.

Client-side

Web · Mobile · TV

JavaScript (web and browser SDK), iOS, Android, React Native, Flutter, Roku, AMP, WebView, Pixel.

Server-side

Backends and jobs

Node.js, Python, Go, Java, Scala, Ruby, PHP, Rust, .NET, C++, Unity, Lua, CLI.

Third party

Webhooks

Adjust, Iterable, Mailgun, Mandrill, SendGrid, Zendesk, or any Iglu-compatible GET or POST request.

Studio

Snowtype

Generate typed tracking SDKs from your tracking plan to catch tracking errors at compile time.

04Attach entities with enrichments

Enable any of these in your pipeline configuration. Each enrichment runs in stream and executes in a fixed order, so results are deterministic.

05Where data lands

Stream loaders write to your warehouse, lake, or both — without an external ETL step. Real-time forwarders push the same events to downstream tools.

Warehouses

Snowflake · BigQuery · Redshift · Databricks

Schemas auto-evolve as you ship new tracking. Events arrive within seconds of being tracked.

Lakes

S3 · GCS · ADLS / OneLake

Open table formats (Iceberg, Delta, Hudi). Query with Athena, Spark, or your warehouse.

Forwarding

GTM Server-Side and native

Forward enriched events in real time to ad platforms, CDPs, and marketing tools.

Reverse ETL

Activation

Push modeled warehouse audiences back to operational tools.

06Signals — real-time attributes

Signals computes user attributes from the event stream as events flow, and serves them to your application through a low-latency API.

  • Provide real-time behavioral data to chatbots and agentic applications
  • Power dynamic pricing, recommendations, and adaptive interfaces
  • Trigger interventions automatically when users meet defined criteria
  • Combine streaming attributes with attributes derived from the warehouse
